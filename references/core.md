# 核心协议

## 通讯规范

- 请求方式：`POST`
- Content-Type：`application/x-www-form-urlencoded`
- 字符编码：UTF-8
- 测试环境：`http://api2uat.lfwin.com`
- 生产环境：`http://api2.lfwin.com`
- 测试环境只支持测试 `apikey=00014005`，只支持当月订单，并且不发送异步通知。

## MD5 签名

步骤：

1. 取所有准备提交的参数，排除 `sign`。
2. 如果某个参数会被提交，即使值为空也要参与签名。
3. 按字段名 ASCII 升序排序。
4. 按 `key=value&key=value` 拼接。
5. 末尾追加 `&signkey=<商户signkey>`。
6. 对原始字符串计算 MD5，结果转为 32 位小写十六进制。
7. 把结果作为 `sign` 提交，`signkey` 不提交。

示例：

```text
请求参数：
service=pay.alipay.qrcode&apikey=00014005&money=0.01&nonce_str=123456

待签名串：
apikey=00014005&money=0.01&nonce_str=123456&service=pay.alipay.qrcode&signkey=punr8ucu

sign：
e2bc07b313c49f166ec03796370d8eb2
```

## 验签

响应和异步通知使用同一套规范：

- 只移除 `sign`，其他字段都参与验签。
- 不要写死字段列表，平台可能新增字段。
- 如果运行时已把数组/对象解析成结构化数据，验签前要转为 JSON 字符串。
- `notify_url` 参与签名前要 URL Decode。
- `status != 10000` 时，LFWin 通常返回未签名的失败信息；不要因为失败响应缺少 `sign` 就误判客户端错误。

## RSA

- 请求签名：字段排序和拼接方式与 MD5 相同，但不追加 `signkey`，使用商户 RSA 私钥签名。
- 响应/通知验签：使用平台/服务端 RSA 公钥验签。
- 不能用商户 RSA 公钥验签平台响应。
- 只有商户 RSA 私钥和平台 RSA 公钥都配置完成时，才使用 `sign_type=RSA`。

## 重要状态码

| status | 含义 |
|---|---|
| `10000` | 成功 |
| `1002` | 暂无记录 |
| `3000` | apikey 错误 |
| `3010` | 请求参数错误 |
| `3030` | 订单号格式错误 |
| `3040` | 订单号不存在 |
| `3050` | 请求过于频繁 |
| `3060` | 时间参数不支持跨月 |
| `3900` | 验签失败 |
| `4000` | 接口返回错误 |
| `4001` | 退款处理中 |
| `4002` | 商户余额不足，退款失败 |
| `4920` | 订单待支付，无法退款 |
| `4960` | 订单已关闭 |
| `4970` | 分账订单无法调用普通退款或撤销 |
| `9999` | 系统错误 |

## 工程约定

- 默认用 MD5，除非商户配置明确要求 RSA。
- 签名逻辑做成统一 helper，同时服务请求签名、响应验签和通知验签。
- 待签名串使用原始值；HTTP 表单提交时可以由客户端按表单协议编码传输。
- 金额在接口边界保持字符串，单位为元。
